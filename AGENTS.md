# MADCORE Gena Agent Rules

Этот файл задает обязательную точку входа для всех следующих сессий по проекту `MADCORE Gena`.

## Язык общения

- Планы, отчеты, объяснения, handoff-документы и checkpoint-файлы писать понятным русским техническим языком.
- Английский оставлять только для команд, путей, имен файлов, переменных, функций и кода.
- Не использовать суржик и необъясненные англоязычные ярлыки в обычном тексте.

## Что это за проект

- Проект: отдельный сайт `MADCORE Gena`
- Локальный путь: `/home/max/MADCORE RF`
- Целевой remote: `git@github.com:MaxCorpOrg/MADCORE-Gena-website.git`
- Рабочий production-домен:
  - сайт: `https://madcore.site`
  - `www`: `https://www.madcore.site`
- Старый временный домен `gena.madcore-kavkaz.ru` больше не считается рабочим адресом и должен только переводить трафик на `https://madcore.site`
- Matomo по этой точке использует общий хост `https://analytics.madcore-kavkaz.ru`, но для `MADCORE Gena` нужен отдельный `site id`.

## Обязательный маршрут входа

Перед любой содержательной работой последовательно прочитать:

1. `AGENTS.md`
2. `docs/START_HERE_FOR_NEW_CHAT.md`
3. `docs/AGENT_CHECKPOINT.md`
4. `docs/PROJECT_STATUS.md`
5. `docs/PROJECT_CHECKPOINT.md`
6. `README.md`
7. `docs/DEPLOYMENT.md`
8. `ARCHITECTURE.md`
9. `NEXT_STEPS.md`
10. последний файл из `docs/checkpoints/`, если каталог не пуст

Только после этого можно переходить к коду, серверу и правкам.

## Обязательный старт

Перед любой значимой задачей:

1. Выполнить `git status -sb`, `git log --oneline -5`, `git remote -v`, если git уже инициализирован.
2. Сверить текущий checkpoint в `docs/PROJECT_STATUS.md` и `docs/PROJECT_CHECKPOINT.md`.
3. Если задача касается VPS, домена, DNS, SSL, production или proxy:
   - сначала прочитать `/home/max/SERVWER 2/api.txt`;
   - не переносить его содержимое в git и документацию.
4. Если задача касается Яндекс.Метрики, Директа, токенов и рекламных API:
   - использовать локальные материалы из `/home/max/АПИ/Яндекс АПИ`;
   - не коммитить секреты.
5. Не начинать новую ветку идей, если в checkpoint уже зафиксирован следующий приоритет.

## Критичные ограничения

- Исходный сайт в `/home/max/MADCORE` не менять без явной необходимости.
- `MADCORE Gena` должен оставаться отдельным проектом:
  - отдельный репозиторий;
  - отдельное окружение;
  - отдельные идентификаторы аналитики;
  - отдельный бот для заявок;
  - отдельная точка развёртывания `/opt/madcore-gena`.
- Не возвращать проект к старым значениям `madcore-kavkaz.ru`, старому номеру или старым рекламным `cmp`.
- Не возвращать проект к использованию `gena.madcore-kavkaz.ru` как рабочего домена.
- Не коммитить `.env*`, ключи, токены, пароли и runtime-мусор.

## Документационная дисциплина

- После каждой значимой задачи обновлять:
  - `docs/PROJECT_STATUS.md`
  - `docs/PROJECT_CHECKPOINT.md`
- При завершении этапа создавать dated checkpoint в `docs/checkpoints/`.
- Если менялись окружение, домен, аналитика, кнопки, bot-flow, UTM или выкладка, это обязательно отражать и в документации, и в checkpoint.

## Проверка перед завершением

- По возможности запускать:
  - `npm run lint`
  - `npm run build`
- Если затронуты deployment-скрипты:
  - `bash -n` для измененных `.sh`
- Если инициализирован git и пользователь просит зафиксировать результат, делать осмысленный локальный commit только после обновления документации.
