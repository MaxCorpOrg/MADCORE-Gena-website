# MADCORE Architecture Review

Обновлено: `2026-05-11`

## Назначение проекта

`MADCORE` остается одностраничным продуктовым сайтом для приема рекламного трафика и заявок на консультацию. Проект не является магазином, каталогом или CRM.

## Текущая схема

- Frontend: `Next.js App Router + React + TypeScript + Tailwind`.
- Runtime: Docker stack `postgres + app + nginx` на Hostkey VPS `151.247.197.153`; Docker build использует `package-lock.json` через `npm ci`.
- Production URL: `https://madcore-kavkaz.ru`.
- DNS: TimeWeb, apex и `www` направлены на Hostkey; `www` редиректит на apex.
- SSL: Let's Encrypt, renewal через `webroot` и `madcore-certbot-renew.timer`.
- Данные: PostgreSQL через Prisma, таблицы `leads`, `clicks`, `events`.
- Уведомления: Telegram-only inbox через `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`.
- WhatsApp: кнопка на сайте остается внешним каналом связи; WhatsApp Cloud API код сохранен как неактивная заготовка.

## Поток рекламного трафика

1. Рекламная ссылка ведет на `/go` с `src`, `cmp`, `cr`, `click_id` и UTM-метками.
2. `/go` записывает клик в `clicks`, проводит базовую антифрод-проверку и сохраняет метки в cookie.
3. Нормальный трафик редиректится на `/`, подозрительный - на `/safe`; редиректы строятся от публичного `PUBLIC_BASE_URL`.
4. Клиент отправляет форму на `POST /api/lead`.
5. Сервер валидирует поля, проверяет honeypot, скорость заполнения, IP-rate-limit и дубли телефона.
6. Заявка сохраняется в `leads`.
7. Сервер пишет события `form_submit`, `lead` и асинхронно отправляет уведомление в Telegram.
8. Результат уведомления фиксируется внутренним событием `lead_notify`.

## Готовность к масштабированию

- Прием заявок не зависит от Telegram: если уведомление не отправится, заявка уже сохранена в PostgreSQL.
- Для первого этапа рекламного трафика достаточно Telegram + PostgreSQL/CSV-отчетов, без мини-админки.
- Для роста нагрузки следующий практичный шаг - добавить регулярные отчеты по качеству лидов и backup БД.
- Для долгосрочного развития можно добавить закрытую админку, CRM-интеграцию или очередь уведомлений, но это отдельный этап.

## Риски и контроль

- `npm audit` показывает риск в транзитивном `postcss` внутри `next`; прямой `postcss` зафиксирован на `8.5.14`, force-fix не использовать.
- `/go` smoke-проверка создает тестовый клик с `src=smoke`, это нормально для проверки рекламного маршрута.
- Production-секреты должны храниться только в server `.env`, не в репозитории и не в документации.
- Перед каждым деплоем нужны `npm run lint`, `npm run build`, secret-scan и production smoke.
